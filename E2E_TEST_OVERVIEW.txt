🔷 Catch-up 扫描规则
┌─────────────────────────────────────────────────────────────────────────┐
│ 孤儿年龄：10 分钟（CATCHUP_ORPHAN_AGE_MS）                             │

│ 扫描条件：level=1 AND parent_id IS NULL AND created_at < cutoff       │
│ 触发条件：按 session 分组，COUNT(*) >= L2_SEGMENT_THRESHOLD (default: 5)│

└─────────────────────────────────────────────────────────────────────────┘

🔷 错误分类

永久性错误（不重试）：
  - session_not_found
  - invalid_turn_start_message
  - LLM 400/401/403（模型拒绝）
  - PermanentLLMError

暂时性错误（重试）：
  - session_still_thinking
  - turn_incomplete_mid_tool_use
  - 网络错误、超时、5xx
  - 数据库写入失败

跳过（不重试，不算错误）：
  - insufficient_real_messages (realMessageCount < 2)
  - trivial_turn (assistantText < 200 && no tools)
  - insufficient_l1_summaries (l1_count < 2)
  - no_source_summaries (无 L1 或 L2)

================================================================================
                            常见问题快答
================================================================================

Q: 幂等性如何保证？
A: 通过 idempotencyKey（插入约束）和 singletonKey（pg-boss）双重机制
   - L1: ON CONFLICT (session_id, level, seq_start) DO NOTHING
   - L2/L3: pg-boss singletonKey 确保同一 key 只有一个 pending 任务

Q: 缓存什么时候生效？
A: 仅当 L1 handler 中 status='error_transient' 且 metadata.cached_result 存在
   下次执行同一 userSeq 时调用 getLatestCachedL1Result()，跳过 LLM 调用

Q: 如何确认 L1→L2 被触发？
A: L1 成功后立即调用 enqueueSegmentIfNeeded()，检查：
   ├─ countUnassignedL1() >= threshold
   ├─ boss.send() 被调用且 queue=SUMMARIZE_SEGMENT
   └─ mock 验证 singletonKey=segment:{sessionId}

Q: Catch-up 何时触发？
A: 启动时运行一次，后续每隔 CATCHUP_INTERVAL_MS（默认 1h）运行
   查找：level=1 AND parent_id IS NULL AND created_at < (now - 10min)

Q: 错误什么时候重试？
A: 暂时性错误自动重试（transient），永久性错误直接返回（permanent）
   重试次数和延迟由队列配置决定，指数退避最多延迟 5min(L1)/10min(L2)/15min(L3)

Q: 如何 mock DeepSeek 而不调用真实 API？
A: 见 E2E_TEST_SNIPPETS.md 片段 5
   设置 ctx.deepseekClient.summarizeTurn = vi.fn().mockResolvedValue(mockResponse)
