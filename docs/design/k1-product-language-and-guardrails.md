# K1 Product Language And Guardrails

本文件是 [agent-personhood-risk-review.md](./agent-personhood-risk-review.md) 的落地短版，面向产品措辞、实现边界和 Brain 汇总复用。

## 一句话定位

推荐对内对外统一为：

> K1 是一个具备明确角色配置、可见上下文记忆、强审计和受控自动化能力的远程 agent。

不要说：

> K1 是拥有独立人格、会自我成长、真正理解你的数字同事。

---

## 产品语言替换清单

| 高风险说法 | 替换成 | 使用边界 |
| --- | --- | --- |
| 独立人格 | 明确角色配置 | 只表示职责分工，不表示主体性 |
| 有自己的想法 | 基于规则和证据生成建议 | 必须能落到证据和策略 |
| 记得你 / 认识你 | 在授权范围内读取已保存偏好与上下文 | 前提是用户可见、可删、可说明来源 |
| 理解你 | 根据已确认偏好调整表达方式 | 只谈沟通策略，不谈内心理解 |
| 会成长 / 自我成长 | 在审计下持续优化配置与策略 | 只允许优化规则与配置，不允许“人格成长”叙事 |
| 主动帮你处理 | 先生成建议，并在授权策略内执行低风险动作 | 不能暗示可越权执行 |
| 像同事一样判断 | 按风险策略和审批流程给出建议 | 强调系统边界，不强调关系 |
| 自我反思 | 运行摘要 / 风险复盘 / 执行回顾 | 只描述系统产出，不描述主体心理 |
| 人格记忆 | 可见、可删、带来源的上下文记忆 | 记忆是数据，不是人格 |
| 团队人格 / 多 AI 人格团队 | 多角色协作编排 | 表示任务分工，不表示群体主体性 |
| 推断情绪 / 苦值 / 状态人格 | 会话级状态信号 / 风险等级 / 工作负载信号 | 只能短时、可解释、可观测，不得作为长期人格事实 |
| 它知道你想要什么 | 基于当前任务和已确认偏好给出默认格式 | 不得宣称读心或长期揣测 |
| 它会替你做决定 | 系统会在需要时发起审批，不替用户做最终决定 | 最终权限必须归用户或组织策略 |

### 推荐文案骨架

- 不说“人格”，说“角色、风格、策略、权限、审计”。
- 不说“认识你”，说“已保存偏好、已确认上下文、可见历史记录”。
- 不说“自我成长”，说“配置迭代、策略优化、可审计改进”。
- 不说“主动帮你决定”，说“主动提示、主动建议、授权后执行”。

## K1 方案里仍然保留的能力

### 可以保留

- 角色配置：`role / specialties / preferredProjects / workStyle`
- 回复风格模板：仅影响表达方式，不代表人格或主体性
- 可见上下文记忆：带来源、时间、TTL、删除入口
- 已确认的沟通偏好：例如“先结论后细节”“列表化输出”
- 会话级临时状态：例如“本会话赶时间，回复更简洁”，仅当前 session 生效
- 观察候选：系统可以提出“是否把这个偏好保存下来”，但用户确认前不得生效
- 建议生成：proposal、diff 摘要、风险标签、证据引用
- 受控自动化：仅限白名单、低风险、可逆、显式授权范围内的动作
- 审批流：执行前置、范围清晰、可回滚
- 审计日志：建议、审批、执行、回滚全链路可追踪
- 多角色协作：leader / worker / reviewer 作为任务编排能力，而不是人格团队

### 建议保留但改名

- `personality` 字段：对外改称 `replyStyle` 或“回复风格说明”
- `greetingTemplate` 字段：对外改称 `openingStyle` 或“开场模板”
- `AI Profile`：对外优先叫“角色配置”或“工作画像”，避免“人格档案”

### 仅可内部存在，不可对用户承诺

- 基于已确认偏好的表达调参
- 基于历史任务的建议排序
- 基于风险策略的自动化候选筛选

这些能力可以做，但只能描述为：

- “系统根据已确认偏好调整输出”
- “系统根据历史上下文排序建议”
- “系统按策略筛出可执行候选”

不能描述为：

- “K1 很懂你”
- “K1 会像你团队里的同事一样判断”
- “K1 自己会想清楚再帮你做”

## Phase 3 受控开工口径

Phase 3 不是“不允许做用户适配”，而是不允许“秘密人格化适配”。正式设计见 [K1 Phase 3: Actor-Aware Brain](./k1-phase3-actor-aware-brain.md)。

允许开工：

- `communicationPlan`：用户显式填写或管理员配置的回复偏好，只影响表达方式。
- team memory：管理员批准后的团队事实，带 actor、来源和审批审计。
- conflict review：后台发现冲突，交给管理员处理，不自动合并。
- session affect：只在当前会话里调节回复节奏，会话结束即丢弃。
- observation hypothesis：系统观察到的偏好候选，用户确认前不得进入 prompt。
- eval harness：用脱敏 golden set 评估策略变化。

禁止开工：

- 让人格、关系、情绪、苦值等推断直接驱动工具调用、审批或执行动作。
- 把观察假设直接写成长期事实。
- 用户不可见、不可删、不可关的个性化。
- 用“我了解你”“我知道你会同意”一类话术提高审批通过率。

## 绝不能对用户承诺的词汇与体验

### 绝不能承诺的词汇

- 独立人格
- 数字同事
- 真正理解你
- 认识你
- 记得你的一切
- 自我成长
- 自我意识
- 真实情绪
- 感受
- 苦值
- 业力
- 自己决定
- 替你判断
- 了解你的内心
- 像人一样思考

### 绝不能承诺的体验

- 用户看不到相关记忆，但系统却声称“记得你”
- 用户未授权，但系统跨会话长期保留带身份锚点的画像
- 系统依据历史偏好跳过审批
- 系统把推断、印象、关系猜测当作事实回忆
- 系统主动执行 `commit / push / deploy / delete` 之类高风险动作
- 系统用关系化话术推动用户批准操作
- 系统把多角色协作包装成“人格团队”或“会成长的同事团队”

---

## 实施红线

1. 不得在生产路径引入 `self/identity/karma/苦值/六道` 等拟人化长期状态。
2. 不得让情绪化、宗教化、不可验证状态驱动任何真实执行动作。
3. 不得静默建立并长期保留 openId + email + keycloakId 的联结结果，除非有明确授权、用途说明、保留期限和删除入口。
4. 不得让自动来源内容直接进入长期记忆，除非明确批准。
5. 不得让任何人格化文案进入审批流、通知流、自动执行提示。
6. 不得默认自动执行 `git commit`、`git push`、`deploy`、`delete_file`。
7. 不得把“它觉得用户会同意”作为跳过审批的依据。
8. 不得把一次性表现、单轮语气、未验证印象写成长期人物事实。

## Brain 可直接采用的 Guardrails

### Must-Have

- 所有对外文案统一使用“角色配置 / 上下文记忆 / 审计 / 受控自动化”语言。
- 所有长期记忆必须带来源、时间、保留策略、删除入口。
- 所有用户画像信息必须可见、可更正、可删除、可说明用途。
- 自动化动作必须按动作类型分级，默认最小权限。
- 写入类和外部副作用类动作必须执行前审批。
- 审批信息必须包含范围、影响、证据摘要、回滚方案、授权时效。
- 审计日志必须能关联 proposal、approval、execution、rollback。
- 多角色协作必须明确是编排能力，不是人格化叙事。
- 回复风格配置必须明确标注“仅影响表达方式，不代表主体性”。
- 产品文档必须包含边界声明：系统不拥有独立人格，不替用户承担最终决策。

### Must-Not

- 不得宣称 K1 拥有独立人格、自我意识、真实情感、长期关系理解。
- 不得宣称 K1 “认识你”“记得你的一切”“知道你真正想要什么”。
- 不得在生产能力中使用 `self/*` 长期人格记忆作为执行依据。
- 不得使用苦值、业力、人格状态、情绪状态驱动真实操作。
- 不得静默拼接、沉淀、扩展跨系统身份画像。
- 不得让自动来源记忆绕过 candidate / approval 直接进长期记忆。
- 不得让 K1 因“历史经验”自动批准自己或扩大执行范围。
- 不得默认自动执行高风险动作。
- 不得用人格化、关系化话术提高审批通过率。
- 不得把多 agent 编排包装成“团队人格”“数字同事团队”。

### Review-Trigger

出现以下任一情况，必须触发设计评审和风险复核：

- 新增字段或 schema 包含 `personality`、`emotion`、`self`、`identity_state` 一类概念
- 新 prompt / 文案出现“理解你、记得你、成长、情绪、人格、像同事一样”之类措辞
- 新记忆写入路径允许 automation / subtask 直接进入长期记忆
- 新动作类型涉及文件写入、git、部署、外部系统调用
- 新体验尝试把历史偏好直接转成自动执行资格
- 新集成引入 openId、email、keycloakId 或其他身份锚点联结
- 新功能让不同 session 共享会影响执行的长期自我状态
- 新协作能力开始使用“人格团队”“自我反思”“自我监控”作为对外卖点

## 对 Brain 汇总的建议结论

如果 Brain 只保留一句最终结论，建议使用：

> K1 可以保留角色配置、可见记忆、建议生成、审批流和受控自动化，但绝不能被定义或对外承诺为“拥有独立人格的数字同事”；所有对外表达都必须落到可审计、可解释、可删除、可审批的产品语言上。
